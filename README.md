# ใช้ AI เขียนโค้ด ไม่ได้แปลว่าลูกค้าเพิ่มงานได้ฟรี

**คู่มือสำหรับ Developer และ Freelancer ยุค AI**  
*วิธีคุม scope ตั้งราคา และรับมือลูกค้าอย่างมืออาชีพ*

---

## เกี่ยวกับเอกสารนี้

เอกสารนี้สรุปแนวทางสำหรับ developer และ freelancer ที่รับงานทำเว็บ ระบบ หรือโปรเจกต์ให้ลูกค้า โดยเฉพาะคนที่ใช้ AI ช่วยเขียนโค้ด และกำลังเจอปัญหาจากความคาดหวังของลูกค้าที่ไม่ตรงกับความเป็นจริงของงานพัฒนา

**เหมาะกับใคร**

- Developer มือใหม่ที่เพิ่งเริ่มรับงาน
- Freelancer ที่รับงานเว็บหรือระบบ
- คนที่ใช้ AI ช่วยเขียนโค้ดและต้องการตั้งขอบเขตงานให้ชัด

**เอกสารนี้ครอบคลุม**

1. สาเหตุที่ลูกค้าเพิ่มงานใกล้ deadline
2. AI ช่วย dev ได้จริงแค่ไหน
3. วิธีตอบลูกค้าเมื่อมี requirement เพิ่ม
4. สิ่งที่ต้องตกลงก่อนรับงาน
5. วิธีประเมินราคางาน
6. ข้อควรระวังสำหรับ freelancer มือใหม่

## ปัญหาที่พบบ่อย

สถานการณ์ที่ developer และ freelancer หลายคนเจอในยุค AI มีลักษณะคล้ายกัน:

> ลูกค้ารู้ว่าเราใช้ AI ช่วยเขียนโค้ด จึงคาดหวังว่างานจะเสร็จเร็วมาก พอใกล้ deadline ลูกค้าส่ง requirement เพิ่มเข้ามา แต่ deadline ไม่ขยับ งบไม่เพิ่ม สุดท้ายคนรับงานต้องทำงานล่วงเวลาและอดนอน

ตัวอย่างคำถามที่พบบ่อย:

> *"ตอนนี้ผมรับงาน freelance ลูกค้าทราบว่าเราใช้ AI ช่วยในการโค้ด พอใกล้ deadline ลูกค้าส่ง requirement มาเพิ่ม ทำให้ต้องอดนอน แต่ deadline ไม่ยืดหยุ่น ควรแก้อย่างไร ควรทำข้อตกลงก่อนรับงานไหม แล้วราคางานควรประเมินอย่างไร ผมเพิ่งหัดรับงาน ยังรับไม่เยอะ"*

ปัญหานี้ไม่ใช่เรื่องของคนใดคนหนึ่ง แต่เป็นปัญหาของ developer ยุค AI โดยทั่วไป สาเหตุหลักมักไม่ใช่แค่ "ลูกค้าไม่ดี" หรือ "dev ไม่เก่ง" แต่เกิดจาก **ขอบเขตงานไม่ชัด** และ **ไม่มีกระบวนการรับมือกับงานเพิ่ม**

## ทำไมต้องมีข้อตกลงก่อนรับงาน

**คำตอบสั้น ๆ: ต้องมีข้อตกลงก่อนรับงานทุกครั้ง**

ไม่ว่าจะเป็นงานเล็ก งานใหญ่ งานคนรู้จัก งานลูกค้าใหม่ งานราคาถูก หรืองานราคาแพง — ควรมีข้อตกลงที่ชัดเจนก่อนเริ่มงานเสมอ

### การรับงาน dev ไม่ใช่แค่การเขียนโค้ด

งานพัฒนาซอฟต์แวร์คือการบริหารหลายด้านพร้อมกัน:

- ขอบเขตงาน (scope)
- เวลา
- ความคาดหวังของลูกค้า
- ความเสี่ยง
- การสื่อสาร
- ความรับผิดชอบต่อผลลัพธ์

### เมื่อไม่มีข้อตกลงตั้งแต่แรก

ลูกค้ามักเข้าใจว่า *"อันนี้ก็รวม อันนั้นก็รวม อันนี้เพิ่มนิดเดียว อันนี้แก้แป๊บเดียว"*

แต่ในมุม developer คำว่า **"นิดเดียว" มักไม่ได้นิดเดียวจริง**

| ลูกค้าบอก | ความจริงฝั่ง developer |
|-----------|-------------------------|
| เพิ่มปุ่มหนึ่งปุ่ม | อาจต้องแก้ database |
| เพิ่ม field หนึ่ง field | อาจต้องแก้ form, validation, API, UI, export, report |
| เพิ่ม role admin | อาจกระทบ permission ทั้งระบบ |
| เพิ่ม payment | อาจกระทบ flow ทั้งหมด |
| เพิ่ม dashboard | อาจต้องเขียน query ใหม่ ทำ chart ใหม่ ทำ responsive ใหม่ |

**หลักการสำคัญ:** คำว่า "เพิ่มนิดเดียว" ต้องถูกประเมินก่อน ไม่ใช่รับปากทันที

## Scope Creep คืออะไร

ในงาน project management ปัญหานี้เรียกว่า **Scope Creep** — คืองานที่เพิ่มเข้ามานอกขอบเขตเดิมเรื่อย ๆ จนโปรเจกต์บานและควบคุมยาก

### ตัวอย่างสถานการณ์

| ช่วงเวลา | สิ่งที่เกิดขึ้น |
|----------|----------------|
| ตอนแรก | ตกลงกันว่าทำเว็บ 5 หน้า |
| ผ่านไปสักพัก | ลูกค้าขอเพิ่มหน้า admin |
| ใกล้ส่งงาน | ลูกค้าขอเพิ่มระบบแจ้งเตือน |
| ก่อน deploy | ลูกค้าขอเพิ่มรายงาน |
| ทุกครั้ง | deadline เดิม งบเดิม |

นี่คือ scope creep ในรูปแบบที่พบบ่อย

### สาเหตุที่โปรเจกต์พัง

โปรเจกต์ล้มเหลวจาก scope creep มักไม่ได้เกิดจากสาเหตุเดียว:

- ไม่ใช่เพราะ developer ไม่เก่งเสมอไป
- ไม่ใช่เพราะ AI ไม่ช่วย
- ไม่ใช่เพราะลูกค้าไม่ดีเสมอไป

**แต่เพราะขอบเขตงานไม่ชัด และไม่มี process ในการรับมือกับงานเพิ่ม**

คำว่า "ขอเพิ่มนิดเดียว" ที่ดูเหมือนใช้เวลาไม่กี่ชั่วโมง บ่อยครั้งกลายเป็นงานเพิ่ม 3–5 วัน เมื่อต้องแก้หลาย layer ของระบบและทดสอบผลกระทบ

## AI ช่วยได้ แต่ไม่ได้ลบความรับผิดชอบ

ลูกค้าหลายคนเข้าใจว่า *"ถ้าใช้ AI ก็น่าจะทำเร็วขึ้น"* ซึ่งเป็นความเข้าใจที่อธิบายได้ เพราะข่าวและโฆษณาเกี่ยวกับ AI เขียนโค้ด AI สร้างเว็บ และ AI generate app มีมากในช่วงนี้

### สิ่งที่ AI ทำได้และทำไม่ได้

| AI ทำได้ | AI ทำไม่ได้ |
|----------|-------------|
| ช่วยสร้าง code | รับผิดชอบ code แทน developer |
| ช่วยเสนอ solution | เข้าใจ business context เท่าคนรับงาน |
| ช่วย debug | รับประกันว่า bug จะหมด |
| ช่วยเขียนเร็วขึ้นในบางส่วน | ทำให้ project ทั้งก้อนเสร็จเร็วขึ้นโดยไม่มีความเสี่ยง |

### งาน dev จริง ๆ ไม่ได้มีแค่เขียนโค้ด

งานพัฒนาซอฟต์แวร์ครอบคลุมหลายขั้นตอน:

- วิเคราะห์ requirement
- ออกแบบ architecture
- ออกแบบ database และ API
- พัฒนา frontend และ backend
- จัดการ auth และ security
- deploy และ test
- แก้ bug และ feedback
- สื่อสารกับลูกค้า
- จัดการ edge case และ production issue

**คนที่ต้องรับผิดชอบคุณภาพงานสุดท้ายคือ developer ไม่ใช่ AI**

### วิธีอธิบายให้ลูกค้าเข้าใจ

ประโยคที่แนะนำ:

> *"ใช่ครับ ผมใช้ AI เป็นเครื่องมือช่วยให้ทำงานเร็วขึ้นในบางส่วน แต่ทุกฟีเจอร์ยังต้องผ่านการออกแบบ ตรวจสอบ ทดสอบ และแก้บัคเหมือนเดิม เพราะคนที่รับผิดชอบคุณภาพงานสุดท้ายคือผม ไม่ใช่ AI"*

ประโยคนี้ไม่ปฏิเสธการใช้ AI แต่ช่วยวางความคาดหวังให้ถูกต้อง

## แยกให้ชัด: Scope, Bug และ Change Request

Freelancer มือใหม่ควรเข้าใจความแตกต่างของ 3 คำนี้ให้ชัด เพราะเป็นพื้นฐานของการสื่อสารกับลูกค้า

### 1. Scope — ขอบเขตงานที่ตกลงกันไว้ตั้งแต่แรก

**ตัวอย่าง:**
- ทำ landing page 5 section
- ทำระบบ login/register
- ทำ dashboard admin
- ทำ CRUD สินค้า
- ทำ responsive
- deploy ขึ้น server

สิ่งที่เขียนไว้ใน scope คือสิ่งที่ต้องทำตามข้อตกลง

### 2. Bug — สิ่งที่อยู่ใน scope เดิม แต่ทำงานผิด

**ตัวอย่าง:**
- login ไม่ได้
- กด save แล้ว error
- ข้อมูลไม่เข้า database
- responsive เพี้ยน
- ปุ่ม submit ใช้ไม่ได้
- API response ไม่ตรงที่ตกลงไว้

Bug ควรได้รับการแก้ไขตามระยะเวลารับประกันที่ตกลงกันไว้

### 3. Change Request — สิ่งใหม่ที่ไม่ได้อยู่ใน scope เดิม

**ตัวอย่าง:**
- เพิ่มระบบ coupon
- เพิ่ม export Excel
- เพิ่ม role admin, editor, user
- เพิ่มระบบแจ้งเตือน
- เพิ่ม payment gateway

**นี่ไม่ใช่ bug แต่เป็นงานเพิ่ม**

### หลักการของงานเพิ่ม

งานเพิ่ม (Change Request) ควรมีผลกับอย่างน้อยหนึ่งอย่างเสมอ:

- เพิ่มงบ
- เพิ่มเวลา
- ลด scope เดิมบางส่วนออก

ถ้าเพิ่มงาน แต่ไม่เพิ่มงบ ไม่เพิ่มเวลา และไม่ลด scope เดิมเลย — นั่นไม่ใช่การบริหารโปรเจกต์ แต่เป็นการให้ developer แบกรับความเสี่ยงทั้งหมด

## วิธีตอบลูกค้าเมื่อมี requirement เพิ่ม

### สิ่งที่ไม่ควรทำ

อย่าตอบ *"ได้ครับ"* ทันทีโดยไม่ประเมิน เพราะคำตอบนี้มักเปิดทางให้ scope ขยายโดยไม่มีขอบเขต

### สิ่งที่ควรทำ

ตอบแบบนี้:

> *"ทำเพิ่มได้ครับ แต่ส่วนนี้เป็น requirement ใหม่นอก scope เดิม ผมขอประเมินผลกระทบเรื่องเวลาและค่าใช้จ่ายก่อน"*

จากนั้นเสนอทางเลือกให้ลูกค้า **3 ทาง**:

| ทางเลือก | รายละเอียด |
|----------|------------|
| **1. แยก phase** | ส่ง scope เดิมให้ทัน deadline ก่อน แล้ว requirement ใหม่ทำเป็น phase 2 |
| **2. เพิ่มเวลาและงบ** | รวม requirement ใหม่เข้าไปในรอบปัจจุบัน โดยขยาย deadline และเพิ่มค่าใช้จ่าย |
| **3. แลก scope** | ถ้าต้องการ deadline เดิม ต้องตัดฟีเจอร์เดิมบางอย่างออก เพื่อเอาเวลามาทำฟีเจอร์ใหม่ |

**Freelancer มืออาชีพไม่ใช่คนที่รับทุกอย่าง แต่คือคนที่ประเมินผลกระทบก่อนรับปาก**

### ตัวอย่างบทสนทนา

**ลูกค้า:** ขอเพิ่มระบบแจ้งเตือนอีกนิดนึงได้ไหมครับ น่าจะไม่นาน

**Developer:**

> ได้ครับ ขอประเมินก่อน เพราะส่วนนี้อยู่นอก scope เดิม การเพิ่มระบบแจ้งเตือนจะกระทบ backend, database, frontend และการทดสอบ มี 2 ทางเลือก:
>
> 1. ส่งระบบตาม scope เดิมให้ทัน deadline ก่อน แล้วระบบแจ้งเตือนทำเป็น phase 2
> 2. เพิ่มเวลาอีก 3 วัน และเพิ่มค่าใช้จ่ายตามฟีเจอร์ใหม่ เพื่อรวมเข้าไปในรอบนี้

วิธีนี้ไม่ปฏิเสธลูกค้า ไม่ใช้อารมณ์ และช่วยให้ลูกค้าเห็นว่างานเพิ่มมีต้นทุนจริง

## 10 เรื่องที่ต้องตกลงก่อนเริ่มงาน

ก่อนรับงาน ควรตกลงอย่างน้อย 10 เรื่องต่อไปนี้

### 1. Scope of Work

เขียนให้ชัดว่าจะทำอะไรบ้าง

**ตัวอย่าง:**
- ทำเว็บไซต์บริษัท 5 หน้า: Home, About, Services, Blog, Contact
- รองรับ responsive
- มีฟอร์มติดต่อ
- เชื่อม Google Map
- deploy ขึ้น hosting

### 2. Out of Scope

เขียนด้วยว่าอะไร**ไม่รวม**ในราคา

**ตัวอย่าง:**
- ไม่รวม logo design
- ไม่รวม copywriting
- ไม่รวม hosting/domain
- ไม่รวมระบบ payment
- ไม่รวม maintenance รายเดือน
- ไม่รวม feature ที่ไม่ได้ระบุในเอกสารนี้

**ทำไมต้องเขียนสิ่งที่ไม่รวม?** เพราะถ้าไม่เขียน ลูกค้าอาจเข้าใจว่ารวมทุกอย่าง

### 3. Timeline

กำหนดวันเริ่ม วันส่ง draft และวันส่ง final ให้ชัด

### 4. Milestone

แตกงานเป็นช่วง ๆ เพื่อติดตามความคืบหน้า

| Milestone | รายละเอียด |
|-----------|------------|
| 1 | สรุป requirement + wireframe |
| 2 | ทำ frontend |
| 3 | ทำ backend |
| 4 | test + deploy |
| 5 | handover |

### 5. Payment

ไม่ควรเริ่มงานใหญ่โดยไม่มีมัดจำ

**ตัวอย่าง:**
- มัดจำ **40%** ก่อนเริ่ม
- จ่าย **30%** หลังเสร็จ milestone กลาง
- จ่าย **30%** ก่อนส่งมอบ final หรือก่อน deploy production

### 6. Revision

กำหนดจำนวนรอบแก้ เช่น แก้ minor revision ได้ **2 รอบ**

**Revision ≠ Feature ใหม่**

| ประเภท | ตัวอย่าง |
|--------|----------|
| Revision | แก้สี แก้ข้อความ แก้ spacing |
| Feature ใหม่ | เพิ่ม login, dashboard, payment |

### 7. Bug Fixing Period

**ตัวอย่าง:** หลังส่งมอบ รับประกันแก้ bug จาก scope เดิม **14 วัน**

**ไม่รวม:**
- bug ที่เกิดจากลูกค้าแก้ code เอง
- requirement ใหม่
- server ที่ลูกค้าเปลี่ยน config เอง
- plugin หรือ third-party ที่เปลี่ยน policy

### 8. Change Request

requirement ใหม่หลังเริ่มงาน ต้อง**ประเมินเวลาและค่าใช้จ่ายใหม่ก่อนเริ่มทำ**

### 9. Communication Channel

ตกลงช่องทางสื่อสาร เช่น Line, Email, Trello, Notion, Upwork

**ควรสรุปทุกอย่างเป็นลายลักษณ์อักษร** ไม่ใช่คุยปากเปล่าอย่างเดียว

### 10. Acceptance Criteria

กำหนดว่าแบบไหนถือว่างานเสร็จ

**ตัวอย่าง:**
- ลูกค้าตรวจแล้ว approve
- ระบบทำงานตาม feature list
- deploy แล้วใช้งานได้
- ส่ง source code แล้ว
- ส่งคู่มือใช้งานแล้ว

ถ้าไม่มี acceptance criteria งานมักไม่จบง่าย ๆ เพราะลูกค้าอาจรู้สึกว่ายังไม่เสร็จตลอดเวลา

## วิธีคิดราคางานให้สมเหตุสมผล

Freelancer มือใหม่มักคิดราคาผิด เพราะคิดแค่ *"ใช้เวลาเขียนโค้ดกี่ชั่วโมง"* แต่งาน dev มีหลายขั้นตอนนอกจากการพิมพ์โค้ด

### เวลาที่ต้องคิดรวม

| หมวด | รายการ |
|------|--------|
| วางแผน | อ่าน requirement, คุยกับลูกค้า, ออกแบบ flow, ออกแบบ database |
| พัฒนา | frontend, backend, เชื่อม API |
| ทดสอบ | test, debug, แก้ responsive |
| ส่งมอบ | deploy, เขียนเอกสาร, แก้ feedback, support หลังส่งมอบ |
| สำรอง | buffer สำหรับปัญหาที่ไม่คาดคิด |

### สูตรคิดราคา

```
ราคางาน = ชั่วโมงทั้งหมดที่ประเมิน × rate ต่อชั่วโมง + buffer
```

**ตัวอย่างพื้นฐาน:**
- ประเมินว่างานใช้ **40 ชั่วโมง**
- rate **500 บาท/ชั่วโมง**
- ต้นทุนเวลา = **20,000 บาท**
- บวก buffer **20–50%** ตามความเสี่ยง

| สถานการณ์ | Buffer แนะนำ |
|-----------|-------------|
| งานชัด / ลูกค้าชัด / scope ชัด | 20% |
| งานไม่ชัด / requirement แกว่ง / ลูกค้าตอบช้า / deadline เร่ง | 40–50% |
| งานด่วนมาก | ควรมี **rush fee** หรือค่าเร่งด่วน |

งานเร่งไม่ใช่แค่ทำเร็วขึ้น แต่กินพลัง เพิ่มความเสี่ยง bug และลดเวลาพัก

### ตัวอย่าง: ระบบจองคิว

| รายการ | ชั่วโมง |
|--------|---------|
| คุย requirement | 3 |
| ออกแบบ database | 3 |
| ทำ frontend | 10 |
| ทำ backend | 12 |
| ทำ admin | 8 |
| test และ debug | 8 |
| deploy | 3 |
| แก้ feedback | 5 |
| **รวม** | **52 ชั่วโมง** |

```
52 × 500 = 26,000 บาท
บวก buffer 30% = 33,800 บาท
→ เสนอราคา 34,000 หรือ 35,000 บาท
```

การคิดราคาแบบนี้ใช้เหตุผล ไม่ใช่ความรู้สึกว่า *"น่าจะสัก 5,000 พอ"* — เพราะถ้าคิดถูกเกินไป คนที่เจ็บคือคนรับงานเอง

## ใช้ AI แล้วควรลดราคางานไหม

**คำตอบ: ไม่จำเป็นต้องลดราคาเพราะใช้ AI**

> **ลูกค้าไม่ได้จ่ายเงินเพื่อซื้อวิธีที่เราใช้ทำงาน — ลูกค้าจ่ายเงินเพื่อซื้อผลลัพธ์**

| อาชีพ | ลูกค้าจ่ายเพราะ |
|-------|----------------|
| ช่างภาพ | ภาพสุดท้ายออกมาดีไหม (ไม่ใช่กล้องรุ่นไหน) |
| Designer | งาน design ใช้งานได้จริงไหม (ไม่ใช่ใช้ Photoshop หรือ Figma) |
| Developer | ระบบใช้งานได้จริงไหม ปลอดภัยไหม ดูแลต่อได้ไหม ส่งมอบตรงไหม |

**AI เป็นเครื่องมือของ developer** ถ้าใช้ tool เก่งขึ้นและส่งงานดีขึ้น นั่นคือ productivity ไม่ใช่เหตุผลที่ต้องลดราคาตัวเองเสมอไป

**ข้อควรซื่อสัตย์:** ถ้างานบางอย่าง AI ช่วยให้ทำเร็วขึ้นมากและความเสี่ยงต่ำ อาจเสนอราคาที่แข่งขันได้มากขึ้นได้ แต่ไม่ควรลดจนลืมคิดค่า review, test, debug และความรับผิดชอบ

## อย่ารับทุกอย่างเพราะกลัวเสียลูกค้า

Freelancer มือใหม่มักกลัวเสียลูกค้า จึงตอบ "ได้ครับ" กับทุกคำขอ:

| ลูกค้าขอ | ผลที่ตามมา |
|----------|-----------|
| เพิ่มงาน | scope บาน |
| ลดราคา | กำไรหาย |
| เร่ง deadline | งาน rush, bug เยอะ |
| แก้ไม่จำกัด | งานไม่จบ |

**ผลลัพธ์:** ได้งานจริง แต่เสียสุขภาพ เวลา กำลังใจ และอาจเสียชื่อเสียง

**การปฏิเสธบางอย่างอย่างมืออาชีพไม่ได้ทำให้ดูแย่ — แต่ทำให้ดูเป็นมืออาชีพ**

**ตัวอย่างประโยค:**

> *"ไม่แนะนำให้เพิ่มฟีเจอร์นี้ก่อน deadline เพราะจะเพิ่มความเสี่ยงเรื่อง bug แนะนำให้ส่งเวอร์ชันแรกให้เสร็จก่อน แล้วค่อยทำฟีเจอร์นี้ใน phase 2 จะปลอดภัยกว่า"*

**Developer ที่ดีไม่ใช่คนที่ทำตามทุกคำสั่งเสมอไป แต่คือคนที่ช่วยลูกค้าตัดสินใจให้โปรเจกต์สำเร็จ**

```
ก่อนเริ่มงาน ขอสรุปขอบเขตงานดังนี้

## Scope งาน

- ทำระบบอะไรบ้าง
```

## ถ้าอยู่กลางโปรเจกต์แล้วลูกค้าเพิ่มงาน ควรทำอย่างไร

เมื่ออยู่กลางโปรเจกต์และลูกค้าเพิ่ม requirement ใกล้ deadline ให้ทำตาม 5 ขั้นตอน:

| ขั้นตอน | สิ่งที่ต้องทำ |
|---------|---------------|
| **1. หยุดรับปากทันที** | อย่าตอบ "ได้ครับ" ทันที — บอกว่า "ขอประเมินก่อน" |
| **2. เขียนรายการ requirement ใหม่** | อย่าคุยแบบลอย ๆ — สรุปเป็น list ว่าลูกค้าขอเพิ่มอะไรบ้าง |
| **3. ประเมิน impact** | แต่ละ requirement กระทบอะไร ใช้เวลากี่วัน เสี่ยงตรงไหน ต้องแก้อะไรบ้าง |
| **4. เสนอทางเลือก** | ส่ง scope เดิมก่อน / เพิ่มเวลา+งบ / ตัดบางอย่างออก |
| **5. ให้ลูกค้า confirm เป็นข้อความ** | ต้องมีข้อความยืนยัน เช่น "ลูกค้ายืนยันให้เพิ่มฟีเจอร์นี้ โดยขยาย deadline เป็นวันที่ ... และเพิ่มค่าใช้จ่าย ... บาท" |

**ถ้าไม่มี confirmation เป็นหลักฐาน อย่าเริ่มงานเพิ่ม**

## คำถามที่พบบ่อย (FAQ)

### ถ้าลูกค้าบอกว่า "เพิ่มนิดเดียว" ทำไมต้องคิดเงินเพิ่ม?

> *"เข้าใจครับว่ามองจากฝั่งผู้ใช้งานอาจดูเหมือนเพิ่มเล็กน้อย แต่ฝั่งระบบต้องมีการแก้ code, test และตรวจสอบผลกระทบกับส่วนอื่น เพื่อให้มั่นใจว่าไม่เกิด bug หลังส่งมอบ"*

### ถ้ากลัวลูกค้าไม่จ้างต่อ ควรยอมไหม?

ถ้าเป็นเรื่องเล็กจริง ๆ และไม่กระทบมาก อาจช่วยได้ในเชิง relationship แต่ต้องบอกให้ชัดว่า:

> *"ครั้งนี้ช่วยให้เป็นกรณีพิเศษ แต่ถ้ามี feature เพิ่มหลังจากนี้จะต้องประเมินเป็นงานเพิ่ม"*

**อย่าช่วยแบบเงียบ ๆ จนลูกค้าคิดว่างานเพิ่มคือฟรีเสมอ**

### ควรคิดราคาเป็นรายชั่วโมงหรือเหมาทั้งโปรเจกต์?

| สถานการณ์ | แนะนำ |
|-----------|-------|
| requirement ยังไม่ชัดมาก | รายชั่วโมง (ปลอดภัยกว่า) |
| requirement ชัด / scope ชัด | เหมาราคาได้ |

ไม่ว่าจะคิดแบบไหน **ต้องมี scope และ change request ชัดเจน**

### Freelancer มือใหม่ rate ควรเริ่มยังไง?

- เริ่มจาก**คำนวณเวลาจริงก่อน** อย่าเดาราคาจากตลาดอย่างเดียว
- ตั้ง rate ขั้นต่ำที่รู้สึกว่าคุ้มกับเวลา ความเครียด และความรับผิดชอบ
- ค่อย ๆ ปรับขึ้นเมื่อ portfolio ดีขึ้น ความเร็วดีขึ้น และความมั่นใจมากขึ้น

### Scope บานมักเกิดจากอะไร?

1. requirement ไม่ชัด
2. ไม่กล้าคิดเงินเพิ่ม
3. ลูกค้าเร่ง deadline
4. ประเมินเวลาผิด
5. ไม่มีข้อตกลงตั้งแต่แรก
